## Role Hierarchy

The boilerplate includes a hierarchical role system where higher roles inherit permissions from lower roles:

```
SUPER_ADMIN  ← Highest level (all permissions)
    ↑
  ADMIN      ← Admin panel access
    ↑
REDACTOR     ← Content management
    ↑
MODERATOR    ← Community management
    ↑
  USER       ← Basic authenticated user
    ↑
 GUEST       ← Unauthenticated user
```

<Callout type="warning">
  **Role inheritance:** A `MODERATOR` automatically has all `USER` permissions, an `ADMIN` has all `MODERATOR` + `USER` permissions, etc.
</Callout>

## Role Descriptions

<Tabs defaultValue="guest">
  <Tab value="guest" label="👻 GUEST">

    - View public content
    - Access landing pages
    - Use login/register forms
    - Read documentation

  </Tab>

  <Tab value="user" label="👤 USER">
    **Basic authenticated users**

    **Permissions:**
    - Access dashboard
    - Update own profile
    - Create own content
    - View own data

    **Cannot:** Access admin features or modify other users' data
  </Tab>

  <Tab value="moderator" label="👮 MODERATOR">
    **Community management**

    **Permissions:**
    - All USER permissions +
    - Moderate content
    - Manage comments
    - Handle user reports
    - Basic content management

    **Cannot:** Access admin panel or user management
  </Tab>

  <Tab value="redactor" label="✍️ REDACTOR">
    **Content management**

    **Permissions:**
    - All MODERATOR permissions +
    - Create/edit articles
    - Manage content categories
    - Publish content
    - Content workflow management

    **Cannot:** Access admin panel or user management
  </Tab>

  <Tab value="admin" label="🔧 ADMIN">
    **Administrative access**

    **Permissions:**
    - All REDACTOR permissions +
    - User management
    - System settings
    - Analytics access
    - Billing management

    **Cannot:** Delete organization or access super admin features
  </Tab>

  <Tab value="super-admin" label="👑 SUPER_ADMIN">
    **Complete system access**

    **Permissions:**
    - All ADMIN permissions +
    - System configuration
    - Delete organizations
    - Manage other admins
    - Access all features

    **Can do everything in the system**
  </Tab>

## Role Management

### Assigning Roles

```typescript
// Admin function to assign roles
export async function assignRole(userId: string, newRole: Role) {
  // Verify current user has permission to assign roles
  const currentUser = await getCurrentUser()
  const ability = await definePermissions(currentUser)

  if (!ability.can('manage', 'User')) {
    throw new Error('Insufficient permissions to assign roles')
  }

  // Update user role
  await updateUserRole(userId, newRole)

  // Log role change for audit
  await logRoleChange({
    userId,
    oldRole: user.role,
    newRole,
    assignedBy: currentUser.id
  })
}
```

### Role Hierarchy Validation

```typescript
// Ensure role hierarchy is respected
export function canAssignRole(assignerRole: Role, targetRole: Role): boolean {
  const roleHierarchy = {
    'GUEST': 0,
    'USER': 1,
    'MODERATOR': 2,
    'REDACTOR': 3,
    'ADMIN': 4,
    'SUPER_ADMIN': 5
  }

  const assignerLevel = roleHierarchy[assignerRole]
  const targetLevel = roleHierarchy[targetRole]

  // Can only assign roles lower than or equal to your own
  return assignerLevel >= targetLevel
}
```

## Common Role Scenarios

## Best Practices

<Steps>
<Step title="Start with minimal roles">
  Begin with USER and ADMIN, add others as needed.
</Step>

<Step title="Document role permissions">
  Keep clear documentation of what each role can do.
</Step>

<Step title="Regular role audits">
  Review and update user roles periodically.
</Step>

<Step title="Principle of least privilege">
  Give users the minimum role needed for their tasks.
</Step>
</Steps>
